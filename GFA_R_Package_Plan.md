# Plan for `gfaR`: Refactoring GFA C-program into an R Package

This document outlines the strategic plan to refactor the Genomic Feature Analyzer (GFA) C-program into a high-performance, idiomatic R package named `gfaR`.

## 1. Project Objectives
- **Full Parity**: Ensure all non-B DNA motif finding algorithms (APR, DR, GQ, IR, MR, STR, Z-DNA) are available in R.
- **Performance**: Use `Rcpp` to leverage the existing optimized C algorithms.
- **Idiomatic R**: Provide a clean interface that accepts R strings/vectors and returns `data.frame` or `GRanges` objects.
- **Extensibility**: Standardize internal data structures to allow for future motif additions.

## 2. Technical Stack
- **R Package Framework**: `usethis`, `devtools`.
- **C/R Integration**: `Rcpp`.
- **Documentation**: `roxygen2`.
- **Testing**: `testthat`.
- **Version Control**: Git.

## 3. Phased Implementation Plan

### Phase 1: Environment & Scaffolding
1.  Initialize the R package structure using `usethis::create_package("gfaR")`.
2.  Configure `Rcpp` support with `usethis::use_rcpp()`.
3.  Set up the `src/` directory to host the refactored C code.

### Phase 2: C Code Refactoring (The "Surgical" Update)
1.  **Decouple I/O from Logic**: The current C code is tightly coupled with `main()` and file-based I/O (`read_fasta`, `print_gff`). We will refactor the core `find*` functions to accept sequence pointers (char arrays) and return structured results instead of writing to global arrays.
2.  **Post-Processing & Filtering Logic**:
    - **Overlap Removal**: Refactor `process_repeats.c` (`process_repeatsCentered`, `process_repeatsIncluded`) into Rcpp-compatible functions to handle nested and redundant motifs.
    - **Classification**: Port `is_subset.c` logic to identify "special" motif types (Cruciform, Triplex, Slipped) based on Purine/Pyrimidine content and structural constraints.
    - **Redundancy Management**: Port `is_subset` and related logic to ensure identical motifs aren't double-counted across different finding passes.
3.  **Standardize Data Structures**: Refine the `REP` struct in `gfa.h` to be compatible with R's memory management or map them to Rcpp DataFrames.
4.  **Modularization**:
    - Move `findAPR.c`, `findDR.c`, etc., into `src/`.
    - Create a central `gfa_core.cpp` that acts as the Rcpp bridge.
    - Adapt utility functions (`cdna.c`, `rcdna.c`, `nulls.c`) for internal use.
5.  **Global State Management**: Replace the large global arrays (`dna`, `mrep`, etc.) with dynamically allocated memory or R-managed objects to prevent stack overflows and allow for thread-safe operations in R.

### Phase 3: R Interface Development
1.  Create R wrapper functions in `R/` for each motif type:
    - `find_apr()`, `find_dr()`, `find_gq()`, `find_ir()`, `find_mr()`, `find_str()`, `find_zdna()`.
2.  Develop a high-level master function `gfa_analyze()` that performs all analyses on a given sequence.
3.  Implement input validation for DNA sequences (ACGT checking).

### Phase 4: Documentation & Best Practices
1.  Write Roxygen2 headers for all exported functions, including `@param`, `@return`, and `@examples`.
2.  Create a Package Vignette (`usethis::use_vignette()`) explaining the scientific background and usage.
3.  Include the original publication reference in the documentation.

### Phase 5: Testing & Validation
1.  **Unit Tests**: Create tests for each motif type using `testthat`, utilizing the `gfa_test.fasta` file as the standard input.
2.  **Cross-Validation with Test Assets**: 
    - Extract and use the gold-standard outputs from `test_files.tar` (e.g., `gfa_test_APR.gff`, `gfa_test_DR.tsv`, etc.).
    - Implement automated tests that compare the R package's `data.frame` output against these original GFF/TSV files to ensure 100% bit-for-bit agreement in motif detection, positioning, and classification.
3.  **Edge Case Testing**: Develop synthetic sequences to test boundary conditions (e.g., motifs at the very start/end of a sequence, sequences with only 'N's, and extremely long sequences).
4.  **Memory Safety**: Use `Valgrind` or R's memory profiling to ensure no leaks occur in the `Rcpp` layer during high-volume sequence analysis.
